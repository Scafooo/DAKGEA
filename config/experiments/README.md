# Experiment Configurations

This directory gathers the YAML recipes that drive the multi-stage pipeline (reduction → augmentation → evaluation).
Every file under `config/experiments` describes one experiment run and is consumed by
`experiments/runner/runner.py` when launching batches.

## File layout

- Top-level standalone configs (e.g. `exp_direct.yaml`, `test_*.yaml`) are small scenarios used for CI, smoke tests,
  or interactive debugging.
- The `massive/` subtree contains bulk sweeps automatically generated for specific datasets / ratios.
  You generally **do not** edit those by hand—instead you use them as inputs for large-scale runs.

## Schema overview

Each experiment YAML follows the structure below:

```yaml
experiment:
  name: <string>
  dataset:
    name: <dataset identifier>
    writer: <output writer key>
  reduction:
    method: <registered reducer>
    ratio: <float>
    writer: <writer key>
    eval: <bool>
    save_dataset: <bool>
    save_model: <bool>
  augmentation:
    method: <registered augmenter>
    ratio: <float>
    writer: <writer key>
    eval: <bool>
    save_dataset: <bool>
    save_model: <bool>
  model: <alignment model key>
  seed: <int>
  clear: <bool>
  overwrite_existing: <bool>
```

### Fields in detail

| Field | Description |
|-------|-------------|
| `experiment.name` | Unique identifier for logs, checkpoints, and directory naming. |
| `experiment.dataset.name` | Dataset reader key (e.g. `openea/zh_en`, `hybea/BBC_DB`). |
| `experiment.dataset.writer` | Writer plan used to persist intermediate datasets. |
| `reduction.method` | Key registered in `src.reduction.registry`; typically `random_entities`, `stub`, etc. |
| `reduction.ratio` | Downsampling proportion applied to the raw dataset before augmentation. |
| `reduction.writer` | Writer used after reduction (usually matches dataset writer). |
| `reduction.eval` | Whether to run evaluation immediately after the reduction stage (baseline). |
| `reduction.save_dataset` / `save_model` | Persist reduced datasets or model artefacts. |
| `augmentation.method` | Augmenter key, e.g. `plm`. Set to `stub`/`baseline` to skip augmentation. |
| `augmentation.ratio` | Additional expansion ratio used by the augmenter; semantic depends on the method. |
| `augmentation.writer` | Writer plan for the augmented dataset. |
| `augmentation.eval` | Run alignment evaluation on the augmented dataset. |
| `augmentation.save_dataset` / `save_model` | Persist augmented dataset / trained models (if the augmenter trains one). |
| `model` | Alignment model key registered in `src.alignment_models.registry` (e.g. `bert_int`, `openea`, `hybea`). |
| `seed` | Global seed propagated to reduction/augmentation/model stages for reproducibility. |
| `clear` | If true, previous outputs are deleted before running the experiment. |
| `overwrite_existing` | Controls whether the runner can reuse cached artefacts. |

## Example: Massive sweep entry

`config/experiments/massive/bert_int_aug_red/zh_en_10_03.yaml` configures a BERT-INT pipeline on `openea/zh_en` with
full reduction (ratio=1.0) and PLM augmentation at 30%. Evaluation is requested at both reduction and augmentation
stages, but neither datasets nor models are persisted (`save_*: false`). The final alignment model is `bert_int` with
seed `11037` and `clear: true`, ensuring the workspace is wiped before execution.

## Tips

- Keep reusable presets under the root (e.g., `exp_reduction.yaml`) and copy/adapt them when crafting new runs.
- Use `massive/` configs only as inputs for batch tools; they are verbose by design and generated by scripts.
- When adding new configs, double-check that `method` keys exist in the relevant registries and that writer names
  correspond to entries defined in `experiments/runner/specs.py`.
